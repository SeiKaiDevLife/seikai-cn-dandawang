# 技术细节实现方案

为了实现这个无后端、纯前端直连阿里云 OSS 的 Web 应用，并且兼顾数据的动态更新与安全防护，需要重点规划以下几个技术难点。

## 一、 数据存储与直连通信架构
应用完全抛弃传统后端数据库，转而依赖阿里云 OSS 存储静态资源（图片）和轻量级数据库载体（JSON 索引文件）。

- **数据载体 (`index.json` 或拆分后的分片数据)**：存放在 OSS 中，记录所有发布的图文结构信息。
  - 数据结构示例：包含 `id`, `type` (normal, grid, custom_grid), `title`, `content`, `image_urls`, `layout_config` (自定义九宫格配置), `timestamp` 等。
- **发布更新机制（读改写模型）**：
  1. 页面发布时：前端利用解密出的 OSS 凭证连接阿里云。
  2. 优先将新生成的 WebP 图片 put 上传至 OSS 特定的路径（如 `/images/YYYY/MM/`）。
  3. 通过 OSS GET 获取最新的 `index.json` 文件到前端内存中解析。
  4. 将本次发布生成的全新笔记对象 `unshift` 追加到数组最前面。
  5. 将最新的 JSON 数组字符串化后，再次 put 覆盖上传至 OSS 中的 `index.json`，完成网站内容的更新。

## 二、 双重安全加密实现（防极客设计）
### 1. 第一层：浏览访问层防护
- **目的**：将无关大众阻挡在网页之外。
- **实现方式**：单向哈希（例如 SHA-256 加盐）。
- **逻辑**：在前端代码或环境变量中配置正确的密码哈希值。用户输入查看密码后，前端利用 Web Crypto API 进行相同的哈希计算，对比一致则在 `localStorage` 写入鉴权标记并渲染主体路由。

### 2. 第二层：核心发布防护（AES-256-GCM 高级对称加密）
- **目的**：保护阿里云 `AccessKeyId` 与 `AccessKeySecret` 不被泄露，防止恶意篡改或盗刷 OSS 流量。
- **实现方式**：
  - **脱机加密阶段**：开发者（我们）在本地利用一个高强度二级密码作为密钥，采用 AES-256-GCM 算法将真实的 AK/SK 加密，得到一段 `Ciphertext (密文)` 和 `IV (初始化向量)`。这段密文将作为常量直接写死在前端源码中。
  - **在线解密阶段**：在点击上传操作时，页面弹出二级验证输入框。前端收到用户输入的二级密码后，通过 `crypto-js` 或原生 Crypto API 尝试解密这串常量。
  - **安全保障**：AES-GCM 包含数据完整性验证 (Auth Tag)。一旦密码错误，解密过程会直接抛出失败错误，无法得出错误的字符串。没有正确的二级密码，攻击者拿到前端代码中的密文也毫无用处。

## 三、 纯前端图片压缩与处理链路
为了节省 OSS 空间以及提高移动端加载速度，发布前图片必须经过严格压缩。
- **处理流程**：
  1. 通过 `<input type="file" multiple>` 拦截用户选取的本地文件或手机相册图片。
  2. **Image/Canvas 内存解析**：将 File 对象读取为 DataURL，绘制到 HTML5 `<canvas>` 元素中。
  3. **尺寸控制**：计算原图宽高比，如果较长的一边超过 `2160px`，则按照原比例计算新宽高，并以新尺寸重绘 Canvas。如果长边不超过 `2160px`，则按照原图尺寸绘制。
  4. **格式转换**：调用 `canvas.toBlob(callback, 'image/webp', { quality: 0.85 })` 将绘制完成的画布导出为 WebP 格式的 Blob。
  5. 经过此流程，所有上传到 OSS 的图片均为对齐长边最高 2160px 的高质量 WebP 文件。

## 四、 OSS 图像动态处理与缓存命中优化
为了提升前端加载性能，降低大图（2160px）带来的网络带宽消耗，需要在图片加载时充分利用阿里云 OSS 的图片处理功能（Image Processing）。
- **动态 Resize 机制**：前端在渲染图片前，根据当前设备的屏幕宽度以及图片在网格/瀑布流中所占的物理尺寸，预估出一个大概的像素大小。
- **缓存命中策略（取整分桶）**：为了避免生成形如 `?x-oss-process=image/resize,w_314` 这样极其碎片化的 URL 导致 CDN 或 OSS 缓存无法命中，前端需实现一个“尺寸阶梯（Bucketing）”算法。
  - **实现逻辑**：设置一系列固定的图片宽度档位（如 `300`, `500`, `800`, `1080`, `2160`）。
  - 当前端计算出图片需要 `314px` 宽度时，向上取整匹配到 `500` 档位，请求 URL 拼接参数：`?x-oss-process=image/resize,w_500`。
  - 这种策略既保证了图片绝对清晰（实际加载分辨率稍大于渲染所需分辨率），又最大化了阿里云节点的缓存命中率，大幅节省流量与加载时间。

## 五、 核心排版渲染技术细节
### 1. Masonry 瀑布流算法计算
- **规则**：设定多列（例如移动端2列），新图片永远插入到当前累积高度最小的列中。
- **实现思路**：
  - 维护一个数组 `columnHeights = [0, 0]`。
  - 遍历 JSON 中的帖子数据，针对每一个帖子，寻找 `columnHeights` 中的最小值对应的索引 `minIndex`。
  - 将该帖子 push 到用于渲染第 `minIndex` 列的数据数组中。
  - 根据该帖子头图/九宫格的预估高度累加 `columnHeights[minIndex] += itemHeight`。
  - 需要监听窗口 Resize 事件以触发重新计算与重排。

### 2. 多模式九宫格渲染
- **标准九宫格**：基于 CSS Grid 布局构建。依靠判定图片数量（1-9）挂载不同的 CSS Class（例如：`.grid-layout-5`）。通过设定 `grid-template-columns` 和 `grid-template-rows` 自动流式排列出“上3下2”等异形网格。对于 10+ 张照片，使用 CSS `backdrop-filter: blur(5px)` 在最后一张图片上方覆盖一层，并使用伪元素绝对定位渲染文字 `+N`。
- **自定义九宫格**：
  - 交互上，在前端生成一个虚拟的 3x3 矩阵，允许用户拖拽鼠标（或手指）圈选多个相邻格子合并。
  - 底层数据：合并操作的本质是生成每个区块在 CSS Grid 中的坐标范围（如 `grid-area: 1 / 1 / 2 / 3`，意为起始行1，结束行2，起始列1，结束列3）。
  - 保存发布后，在首页和详情页即可直接通过内联 CSS `style={{ gridArea: '1 / 1 / 2 / 3' }}` 来高保真还原用户拖拽出来的自定义网格布局。

## 六、 Python 脚本初始化辅助 (`_uploader/` 机制)
考虑到纯前端开发前期，需要海量真实的测试数据用于优化瀑布流和九宫格体验。
- **使用场景**：在项目仓库的 `_uploader` 文件夹中编写本地 Python 预处理脚本。
- **功能设计**：
  1. 读取指定本地目录内的生肉照片。
  2. 使用 `Pillow` (PIL) 进行自动裁剪、WebP 压缩（模拟前端行为）。
  3. 按照定义的结构拼装出初始化的全局 `posts.json` 并存放至项目的 `data/` 目录中。
  4. 生成的图片保存至静态目录 `public/`。
  5. 这将允许我们在第一阶段脱离真实的阿里云环境，快速通过本地静态数据实现精美 UI 框架的搭建和排版调试。
